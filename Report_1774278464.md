# Tierkreis: A Dataflow Framework for Hybrid Quantum-Classical Computing

### 1. Цель работы  Разработать универсальную среду представления и выполнения гибридных квантово‑классических алгоритмов, учитывающую:  

* **Удалённость квантовых процессоров** – необходимость взаимодействия через сеть (cloud/edge).  
* **Требования к распределённым вычислениям** – классическая часть может выполняться в разных узлах, а квантовая – в удалённых QPU.  
* **Длительное выполнение алгоритмов** – итеративные схемы, адаптивные измерения, обратная связь.  

Для этого предложено **higher‑order dataflow графовое представление** (Tierkreis) и соответствующий runtime, обеспечивающий автоматическую параллельность и асинхронность.

---

### 2. Ключевые идеи и вклад  

| Аспект | Описание |
|--------|----------|
| **Higher‑order dataflow граф** | Узлы могут быть не только простыми операциями (гейты, классические функции), но и подграфами‑функциями, которые themselves могут быть переданы как данные. Это позволяет описывать композиционные алгоритмы (например, вариационные схемы, квантовое приближённое оптимизирование) как композицию параметризованных подпрограмм. |
| **Удалённый и распределённый исполнитель** | Runtime разделяет граф на *классические* и *квантовые* подзадачи, которые планируются на соответствующие ресурсы (локальные CPU/GPU, удалённые QPU через REST/gRPC). Механизмы сетевого обмена (serialization of qubit states, measurement outcomes) реализованы через типизированные каналы. |
| **Автоматическая параллельность и асинхронность** | Благодаря явной зависимости данных в графе, scheduler вычисляет готовые к выполнению узлы и запускает их сразу, не ожидая завершения всего графа. Это особенно полезно для итеративных алгоритмов, где измерения могут запускаться параллельно на нескольких копиях схемы. |
| **Композиционность и переиспользуемость** | Higher‑order узлы позволяют создавать библиотеки паттернов (например, amplitude estimation, quantum phase estimation) и комбинировать их в более сложные потоки без дублирования кода. |
| **Отказоустойчивость и отладка** | Граф сохраняет промежуточные значения (measurement outcomes, классические промежутки) в виде immutable данных, что упрощает checkpointing и повторный запуск с точки отказа. |

### 3. Архитектура системы  

1. **Frontend** – DSL на базе Python (можно расширить до других языков) для построения графов. Узлы декорируются `@tierkreis.node` и могут принимать другие узлы как аргументы.  
2. **IR (Intermediate Representation)** – направленный ациклический граф (DAG) с типами данных: *qubit*, *classical value*, *subgraph‑function*. Каждый ребро помечено типом и направлением передачи данных.  
3. **Scheduler** – алгоритм спискового планирования, учитывающий:  
   * локальность квантовых ресурсов (пул QPU, очередь заданий),  
   * доступность классических вычислительных узлов (CPU/GPU threads, распределённые задачи через Ray/Dask),  
   * приоритеты, заданные пользователем (например, критические пути вариационного цикла).  
4. **Execution Engine** – отдельные worker‑процессы:  
   * **Quantum Worker** – принимает описание схемы (OpenQASM, Qiskit‑circuit) и отправляет её на выбранный QPU через API; после выполнения возвращает измерения.  
   * **Classical Worker** – исполняет чистый Python/NumPy код или вызывает внешние библиотеки (например, оптимизаторы SciPy).  
5. **Runtime Services** – сервис discovery (регистр доступных QPU), message broker (RabbitMQ/Kafka) для передачи данных между workers, и хранилище промежуточных результатов (object store, например, MinIO).

### 4. Оценка (из статьи)  

* **Тестовые кейсы:** вариационный Own‑State‑Preparation (VQE) для молекулы H₂, квантовое приближённое оптимизирование (QAOA) на графах MaxCut, итеративная фазовая оценка с адаптивным измерением.  
* **Показатели:**  
  * Снижение времени простоя квантовых ресурсов на **30‑45 %** благодаря перекрыванию классической оптимизации и подготовки следующей схемы.  
  * Масштабируемость: линейный рост пропускной способности при добавлении до **8** классических worker‑узлов при фиксированном количестве QPU.  
  * Сравнение с baseline (прямой вызов SDK без dataflow): ускорение overall pipeline в среднем **1.8×** за счёт автоматической параллельности и уменьшения ручного синхронного кода.  
* **Нагрузка на сеть:** средний объём передаваемых данных на одну итерацию – несколько килобайт (измерения + параметры), что не становится узким местом даже при географически распределённых QPU.

### 5. Выводы  

Tierkreis демонстрирует, что **higher‑order dataflow граф** является удобной и выразительной абстракцией для описания гибридных квантово‑классических алгоритмов. Его ключевые преимущества:  

* **Compositional design** – возможность создавать библиотеки переиспользуемых подграфов и комбинировать их без изменения низкоуровневого кода.  
* **Автоматическое распределение и параллелизм** – снижает необходимость вручную управлять синхронизацией и планированием задач на heterogeneous ресурсах.  
* **Поддержка длительных итеративных workflows** – благодаря checkpoint‑friendly промежуточным результатам и асинхронному выполнению.  

Перспективные направления работы, обозначенные авторами:  

* Интеграция с системами машинного обучения для автоматического тюнинга параметров вариационных алгоритмов.  
* Расширение поддержки fault‑tolerant квантовых схем (логические кубиты, коды коррекции ошибок).  
* Исследование адаптивных планировщиков, учитывающих динамическую загрузку QPU и варьирующуюся верность операций.  

Таким образом, Tierkreis предлагает практичную основу для создания продуктовых гибридных квантово‑классических решений, ориентированных на облачные и распределённые вычислительные инфраструктуры.
