### 1. Введение  
Растущая доступность квантовых процессоров в виде облачных сервисов требует новых методов построения гибридных квантово‑классических алгоритмов. Основные вызовы – удалённое выполнение квантовых подпрограмм, необходимость интеграции с распределёнными вычислениями и длительные (часто итерационные) workflows.  

### 2. Постановка задачи  Требуется представление программ, которое:  
- Позволяет композиционно описывать квантово‑классические подзадачи;  
- Отражает естественный способ мышления разработчиков (графы потока данных);  
- Обеспечивает автоматическую параллельность и асинхронность выполнения;  
- Поддерживает удалённый доступ к квантовым ресурсам и интеграцию с облачными/распределёнными системами.  

### 3. Архитектура Tierkreis  

| Компонент | Описание |
|-----------|----------|
| **Dataflow‑graph IR** | Высший‑ордерный ориентированный граф, где узлы – чистые функции (классические или квантовые ядра), а рёбра – потоки данных (тензоры, состояния, классические значения). Позволяет задавать высшие‑ордерные композиции (функции, принимающие функции). |
| **Типовая система** | Статически типизированная, поддерживает квантовые типы (qubit‑регистры, измерения) и классические типы; обеспечивает безопасность composition‑level. |
| **Runtime‑слой** | Выполняет граф в режиме *lazy evaluation*: узел запускается, когда все его входные данные готовы. Поддерживает: <br>• **Автоматическую параллельность** – независимые ветви графа выполняются одновременно; <br>• **Асинхронность** – ожидание результатов удалённых квантовых вызовов не блокирует граф; <br>• **Распределённое планирование** – узлы могут быть отнесены к локальному CPU, GPU или удалённому QPU через абстракцию *device*. |
| **Механизм удалённого вызова** | Для каждого квантового ядра генерируется stub, который сериализует входные данные, отправляет их в облачный QPU, ждет результата (измерения, состояния) и продолжает выполнение графа. |
| **Поддержка long‑running workflows** | Граф может содержать циклы (через высшие‑ордерные конструкции, например, fold/map) и контрольные точки; runtime сохраняет промежуточные состояния, enabling checkpoint‑restart и повторные запуски без потери прогресса. |

### 4. Ключевые свойства  

1. **Compositional design** – благодаря высшему‑ордеру можно строить библиотеки повторно используемых квантовых подпрограмм (например, амплитудное усиление, вариационные ansätze) и комбинировать их в сложные алгоритмы.  
2. **Автоматическая параллельность** – статический анализ графа выявляет независимые подграфы, которые планируются на разных вычислительных узлах без вмешательства программиста.  
3. **Асинхронность** – ожидание квантовых измерений не блокирует выполнение остальных частей графа, что критично для dlouhýchIterations (VQE, QAOA, алгоритмы фазовой оценки).  
4. **Прозрачное удалённое выполнение** – абстракция device скрывает детали сетевого взаимодействия, пользователь работает с тем же IR, независимо от того, где выполняется узел.  
5. **Поддержка облачных и гибридных инфраструктур** –runtime может планировать части графа на локальных кластерах (CPU/GPU), а квантовые подзадачи – на нескольких QPU разных провайдеров, обеспечивая баланс между latency и cost.  

### 5. Пример использования (схематично)  

```
graph G:
  1. state_prep  :: |0>^{\otimes n} -> |psi0>
  2. ansatz      :: (params: Real^m, |psi>) -> |psi'>   # квантовое ядро  3. measure     :: |psi'> -> Real^m   # измерение ожидаемого значения
  4. cost_func   :: Real^m -> Real    # классическая пост‑обработка
  5. optimizer   :: (Real -> Real) -> Real^m  # классический итератор (например, SPSD)
  6. loop        :: optimizer(cost_func ∘ measure ∘ ansatz) -> params_opt
```

Каждый из пунктов 1‑5 представляется узлом в графе; узел 6 – высший‑ордерный оператор, реализующий итерационный цикл. Runtime автоматически распараллеливает оценку `ansatz` для разных наборов параметров (если используется batch‑mode) и асинхронно ждёт результатов измерений, пока optimizer готовит следующий набор параметров.

### 6. Выводы  

Tierkreis предлагает строго типизированный, высший‑ордерный dataflow‑IR, совместно с рантаймом, который решает три основные проблемы гибридных квантово‑классических вычислений: удалённый доступ к квантовым ресурсам, необходимость распределённого и облачного выполнения, а также длительные итерационные workflows. Благодаря явному представлению зависимостей в графе, система достигает автоматической параллельности и асинхронности без дополнительного программистского труда, что делает её перспективной основой для построения библиотек варьирующихся от вариационных алгоритмов до квантовых машинных обучений и квантово‑классических оптимизаторов.
