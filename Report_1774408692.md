# Tierkreis: A Dataflow Framework for Hybrid Quantum-Classical Computing

**Технический отчет**  
*На основе статьи:* **Tierkreis: A Dataflow Framework for Hybrid Quantum‑Classical Computing**  

---

### 1. Аннотация  
Tierkreis – это фреймворк, представляющий гибридные квантово‑классические алгоритмы в виде **высшего порядка графов потока данных (dataflow graphs)**. Система обеспечивает композиционное построение программ, автоматическую параллелизацию и асинхронное выполнение, учитывая удалённый доступ к квантовым процессорам и потребность в распределённых вычислениях в облаке.

---

### 2. Мотивация  
| Проблема | Обоснование в Tierkreis |
|----------|--------------------------|
| **Удалённость квантовых устройств** | Требуется эффективная передача данных и управление задачами через сеть. |
| **Гибридные алгоритмы** | Необходимо сочетать короткоживущие квантовые подпрограммы с длительными классическими вычислениями. |
| **Облачные/распределённые среды** | Поддержка динамического масштабирования и отказоустойчивости. |
| **Длительные исполнения** | Возможность перекрывать вычисления (overlap) и выполнять их асинхронно без блокировки основного потока. |

---

### 3. Архитектурные принципы  1. **Высший‑порядковый граф потока данных**  
   - Узлы – функции (классические или квантовые ядра).  
   - Рёбра – типизированные потоки данных (ква‑биты, классические значения, промежуточные результаты).  
   - Граф может содержать **подграфы‑функции**, что обеспечивает композиционность и повторное использование.

2. **Рантайм‑слой**  
   - Планировщик задач, учитывающий latencies сети и доступность квантовых бэкендов.  
   - Механизмы **асинхронного диспатчинга** и **prefetch‑ing** данных между классическими и квантовыми узлами.  
   - Поддержка **отказоустойчивого выполнения** (checkpoint‑restart) для長running workflows.

3. **Автоматическая параллельность**  
   - Анализ зависимостей в графе позволяет выявлять независимые подграфы и запускать их параллельно на разных вычислительных узлах (CPU, GPU, QPU).  
   - Динамическое балансирование нагрузки в облачной инфраструктуре.

4. **Интерфейс пользователя**  
   - DSL (доменно‑специфичный язык) на базе функционального программирования (например, Haskell‑подобный синтаксис) для построения графов.  
   - Визуальный редактор, отражающий схему алгоритма, что упрощает отладку и совместную работу команды.

---

### 4. Ключевые компоненты  

| Компонент | Функция |
|-----------|---------|
| **Graph Builder** | Парсит DSL, строит типизированный dataflow‑граф, выполняет статический анализ типов и зависимостей. |
| **Scheduler** | Определяет порядок исполнения узлов, учитывая ресурсы (QPU‑время, network latency) и политики приоритетов. |
| **Execution Engine** | Запускает задачи на соответствующих бэкендах (классические серверы, GPU, удалённые QPU), управляет передачей квантовых состояний через сериализацию/десериализацию (например, OpenQASM, Quil). |
| **Monitoring & Debugging** | Логирует прогресс графа, предоставляет трассировку асинхронных событий и визуализацию текущего состояния выполнения. |
| **Persistence Layer** | Хранит промежуточные результаты и checkpoint‑ы для восстановления после сбоев или для долгих запусков. |

### 5. Преимущества по сравнению с существующими решениями  

| Критерий | Tierkreis | Типичные подходы (Qiskit, Pennylane, т.д.) |
|----------|-----------|--------------------------------------------|
| **Композиционность** | Высший‑порядковые подграфы → легкое повторное использование | Обычно линейные скрипты; повторное использование требует ручного копирования кода. |
| **Автоматическая параллельность** | Зависимости выявляются статически → планировщик распределяет задачи | Параллелизм реализуется вручную (multiprocessing, futures) или отсутствует. |
| **Асинхронность** | Неблокирующий обмен данными между классическими и квантовыми узлами | Часто синхронный вызов QPU → простои ожидания. |
| **Облачная/распределённая поддержка** | Встроенная работа с удалёнными бэкендами, учёт сетевых задержек | Требует отдельной оркестрации (Kubernetes, scripts). |
| **Масштабируемость** | Динамический баланс нагрузки, checkpoint‑restart | Ограничено ресурсами одного узла или ручным управлением. |
| **Визуализация** | Граф ↔ алгоритм (один‑к‑одному) | Требует отдельных диаграмм или построения вручную. |

### 6. Выводы  
Tierkreis предлагает **концептуально чистую и практически эффективную** модель для построения гибридных квантово‑классических алгоритмов:  
- представление в виде типизированного dataflow‑графа делает алгоритмы **прозрачными** для проектировщиков и облегчает их визуальный анализ;  
- высший‑порядковый подход обеспечивает **композиционность** и повторное использование сложных подпрограмм;  - встроенный планировщик и исполнитель реализуют **автоматическую параллельность и асинхронность**, что критично для работы с удалёнными квантовыми процессорами и длительными вычислениями в облачных средах.  

Эти свойства делают Tierkreis перспективной основой для разработки следующего поколения гибридных квантовых приложений, в частности, алгоритмов вариационной оптимизации, квантово‑машинного обучения и квантово‑ химических симуляций.

### 7. Направления дальнейших работ  

1. **Интеграция с существующими SDK** – предоставление адаптеров для Qiskit, Cirq, Pennylane и т.п., чтобы пользователи могли постепенно мигрировать既有 код в графовый формат.  
2. **Расширение модели типов** – поддержка квантово‑корректирующих кодов, топологических состояний и ресурсоориентированных типов (например, количество кубитов, коherence time).  
3. **Автоматическое оптимизирование графа** – применение правил преобразования (fusion, loop‑unrolling, удаление мёртвых узлов) для уменьшения количества сетевых круговых обходов.  
4. **Тестирование на реальных гибридных кластерах** – оценка производительности на платформах типа IBM Quantum Network, Azure Quantum, AWS Braket при различных нагрузках и сетевых условиях.  
5. **Инструменты профилирования и отладки** – визуализация критических путей, оценка стоимости передачи квантовых состояний и предложения по размещению задач ближе к QPU.
