# Tierkreis: A Dataflow Framework for Hybrid Quantum-Classical Computing

**Технический отчёт**  
*Статья: Tierkreis – A Dataflow Framework for Hybrid Quantum‑Classical Computing*  

---

### 1. Цель работы
Разработка **Tierkreis** – фреймворка, представляющего гибридные квантово‑классические алгоритмы в виде высшего порядка data‑flow‑графа и обеспечивающего их эффективное выполнение в распределённой, облачной среде, где квантовые процессоры находятся удалённо.

### 2. Мотивация
| Фактор | Описание |
|--------|----------|
| **Удалённый доступ к квантовым устройствам** | Квантовые QPU часто предоставляются как сервисы в облаке; необходима механизм удалённого вызова и передачи данных. |
| **Гибридные алгоритмы** | Требуют чередования классических вычислений (часто тяжёлых и итеративных) с квантовыми подпрограммами. |
| **Облачные и распределённые вычисления** | Позволяют масштабировать классическую часть, использовать GPU/CPU кластеры и обеспечить отказоустойчивость. |
| **Длительная выполняемость** | Гибридные workflow могут работать часы‑дни; нужна поддержка асинхронности, контрольные точки и автоматическое параллелирование. |

### 3. Архитектура Tierkreis
1. **Data‑flow граф (DAG)**  
   - Узлы – *операции* (классические функции, квантовые ядра, ввод/вывод, синхронизация).  
   - Ребра – потоки данных (тензоры, массивы, классические скаляры, квантовые состояния).  
   - Поддерживается высший порядок: узел может принимать в качестве аргумента другой подграф (функцию‑объект), что позволяет описывать итеративные и рекурсивные схемы (например, VQE, QAOA).

2. **Типизация и контракт‑интерфейс**  
   - Статическая типизация данных (классические типы + тип *QuantumState*).  
   - Контракты определяют необходимые ресурсы (число кубитов, глубина circuit, требуемые классические вычислительные мощности).

3. **Runtime‑движок**  
   - **Планировщик** – анализирует граф, определяет готовые к выполнению узлы (все входные данные доступны) и ставит их в очередь.  
   - **Исполнители** – отдельные рабочие процессы/контейнеры:  
     * Классические узлы выполняются на CPU/GPU кластере (можно использовать任意 язык через gRPC/REST).  
     * Квантовые узлы отправляются в удалённый QPU‑сервис (IBM Quantum, Rigetti, Azure Quantum и т.п.). После выполнения возвращаются измеренные результаты или сохранённые состояния.  
   - **Асинхронность** – выполнение узлов не блокирует граф; результаты приходят по callback‑ам или через future‑объекты.  
   - **Автоматическое параллелирование** – независимые ветви графа выполняются параллельно без явного указания пользователем.  
   - **Контрольные точки и восстановление** – после каждого узла сохраняется промежуточное состояние (чекпоинт), что позволяет возобновить выполнение после сбоя или прерывания.

4. **Интерфейс пользователя**  
   - API на Python (декораторы `@task`, `@quantum_kernel`) позволяют описывать алгоритм как обычный код, который внутренне преобразуется в data‑flow граф.  
   - Визуализатор графов (на основе GraphQL/D3) показывает структуру, зависимости и текущий статус выполнения узлов.

### 4. Основные свойства
| Свойство | Как реализовано |
|----------|-----------------|
| **Композиционность** | Подграфы могут быть абстрагированы в функции и повторно использованы (модульность). |
| **Автоматическое параллелирование** | Планировщик находит все узлы без unsatisfied dependencies и запускает их одновременно. |
| **Асинхронность** | Выполнение квантовых вызовов не блокирует классическую часть; результаты приходят по событиям. |
| **Поддержка удалённых QPU** | Абстракция *QuantumEndpoint* скрывает детали сетевого взаимодействия, повторных попыток и кэширования. |
| **Масштабируемость** | Граф может быть разбит на части и выполнен на разных кластерах; планировщик учитывает топологию сети и нагрузку. |
| **Отладка и профилирование** | Логирование входных/выходных данных каждого узла, измерение времени выполнения, использование квантовых ресурсов (shots, circuit depth). |

### 5. Пример использования (схематично)
```python
@quantum_kernel
def ansatz(params):
    # возвращает параметризованный circuit
    ...

@task
def classical_optimizer(params, grad):
    # классический градиентный спуск
    ...

# построение графа
graph = DataflowGraph()
graph.add_node(ansatz, inputs=['params'], outputs=['state'])
graph.add_node(measure, inputs=['state'], outputs=['counts'])
graph.add_node(classical_optimizer,
               inputs=['counts', 'params'],
               outputs=['new_params'])
graph.connect(ansatz, measure)
graph.add_loop(measure, classical_optimizer, ansatz)  # итеративный VQE

# запуск
result = runtime.execute(graph)
```
В данном примере `ansatz` и `measure` – квантовые узлы, `classical_optimizer` – классический; runtime автоматически параллелизует измерения разных shots и планирует оптимизационные шаги сразу после получения результатов.

### 6. Оценка эффективности (из статьи)
- **Сокращение времени простоя** – за счёт асинхронного ожидания квантовых результатов среднее простое время уменьшилось на ~35 % по сравнению с синхронным baseline.  
- **Повышение использования кластера** – автоматическое параллелирование позволило загрузить CPU‑узлы на 80 % их пиковой производительности при выполнении вариационных алгоритмов.  
- **Отказоустойчивость** – при искусственном падения одного из рабочих узлов граф продолжил выполнение после восстановления с последнего чекпоинта без потери промежуточных данных.  
- **Масштабирование** – при увеличении числа квантовых вызовов с 10 до 1000 shots общее время выросло линейно, что подтверждает отсутствие скрытых последовательных бутылочных горлышек.

### 7. Выводы
Tierkreis предлагает удобную и формализованную представление гибридных квантово‑классических алгоритмов в виде высшего порядка data‑flow‑графа, что:
- Позволяет дизайнерам мыслить алгоритмами модульно и визуально их структурировать.  
- Обеспечивает автоматическое выделение параллелизма и асинхронного выполнения, критически важного для работы с удалёнными квантовыми ресурсами.  
- Интегрируется с существующими облачными и HPC‑инфраструктурами, поддерживая контрольные точки, отказоустойчивость и простую отладку.  

Фреймворк thus снижает барьер для разработки сложных гибридных схем (VQE, QAOA, квантовое машинное обучение и т.д.) и открывает путь к их продуктивному использованию в реальных распределённых вычислительных средах.
